Filters Module
Philosophy

The Filters module is built around a pipeline-driven data transformation approach where input flows through a sequence of reusable, middleware-like stages. It emphasizes composition over direct function chaining, allowing filters to be dynamically resolved and applied via a registry. The design promotes decoupling between filter definition, registration, and execution, enabling flexible and extensible data processing across different contexts.

File Structure

├───filters
│       Filters.php
│       Registry.php
│       StageInterface.php
│       StageList.php
│       StageListInterface.php

Features

- Pipeline-based filtering system
- Middleware-style stage execution
- Registry-driven stage resolution
- Support for both single stages and grouped stage lists
- Batch filtering for multiple inputs
- String and array-based filter definitions
- Automatic stage discovery via directory scanning
- Extensible custom filter stages
- Built-in common sanitization and normalization filters

Interesting / Advanced Parts

- Middleware-style filters (each stage controls flow via $next)
- Registry abstraction (decouples filter naming from implementation)
- StageList concept (multiple filters defined in a single class)
- Directory-based auto-registration of filters
- Reuse of generic pipeline system instead of custom execution logic

Strengths

- Clean separation: definition, registration, execution
- Highly extensible (easy to add custom filters)
- Reusable across different domains (not HTTP-specific)
- Simple and expressive usage ('trim,lowercase')
- Supports both granular and grouped filter definitions

Limitations

- No support for parameterized filters (e.g., max:10)
- Inconsistency between stored method names and pipeline via('process')
- Silent type handling may hide errors
- No structured error/validation feedback system
- Registry lacks namespacing or contextual scoping
